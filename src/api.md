# api 模块技术总结

## 模块定位
`api/` 是平台后端入口，负责 HTTP 接口、鉴权、数据服务、任务状态编排和系统运行时装配。

## 目录概览
- `apps/`：按业务域拆分接口，如知识库、文件、对话、系统、用户、INIS 业务扩展等。
- `db/`：模型、初始化、运行时配置和数据库服务层。
- `services/`：对外部工作流或业务逻辑的服务封装。
- `utils/`：日志、文件、命令、Web 响应等通用能力。
- `ragforge_server.py`：后端启动入口与后台维护任务。

## 核心职责
- 提供 REST 风格接口与登录态访问控制。
- 管理知识库、文档、分块、对话、模型与用户配置。
- 维护任务进度、恢复异常中断任务并清理连接资源。
- 作为 `rag/`、`minerU/`、`graphrag/`、`web/` 的服务中枢。

## 关键实现点
- `ragforge_server.py` 内置数据库重试、后台进度更新、内存监控和周期维护逻辑。
- `api/apps/inis_app.py` 扩展了知识库分类、批次上传与 INIS 业务流程。
- `api/db/services/*` 体现明显的服务层封装，有利于接口和存储逻辑解耦。

## 运行依赖
- Flask 应用栈。
- 数据库、缓存、搜索引擎、对象存储。
- 来自 `conf/` 的运行配置与模型工厂定义。

## 风险与建议
- 配置与外部依赖较多，建议优先通过环境变量或模板文件管理。
- 生产环境需要单独处理真实凭据、密钥与证书，避免把环境专属值写入代码。
